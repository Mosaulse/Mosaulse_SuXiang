# 素缃项目 · AI 开发指南

## 项目概况

VSCode 古典配色主题扩展（`suxiang-theme`），所有颜色均采用[中国传统配色](https://zhongguose.com/)。此卷承华夏色谱千年余韵，取古籍设色之法，化为今人编辑器中可亲可用之配色方案，使开发者于代码之间，犹展古人画卷。

## 设计总纲

### 色彩宗旨

- **凡用色者，必有所本**——每一色名皆有来历，每一色调皆循中国传统色谱，不可臆造。
- **亮暗相生**——亮色主题与暗色主题非简单取反，乃如昼夜轮转、阴阳互生，各自成境而又气韵相通，**相得益彰**。
- **和谐可读**——配色须兼顾色彩和谐、文本可读性，且满足 [WCAG 无障碍对比度](https://www.w3.org/WAI/standards-guidelines/wcag/) 标准（正文文字对比度不低于 4.5:1，大字标题不低于 3:1），确保目力所及，清晰可辨。
- **护眼为本**——暗色主题忌纯黑死白之刺目，亮色主题忌高饱和荧光之灼眼。取沉稳温润之调，久视而不疲，是为上品。
- **繁简有度**——代码高亮颜色控制在 **5 至 7 种**以内，分高频与低频两类。关键字、函数、变量等高频色宜沉稳易辨；字符串、注释等低频色宜柔和退后。色不在多，有层次则灵。

### 典故之制

每一主题皆当有出处。或取自古籍诗文，或源于山水花鸟，或化自器物风物，使主题不仅是一套配色，更是一段可以讲述的故事。例：

- 「素缃」取缃帛之色——缃者，浅黄也，古人以此色绢帛书写经籍，朴素而温雅。
- 主题名、色名若有典故，须在 README 或主题描述中注明出处，使读者知其然亦知其所以然。

## 技术架构

- 所有主题文件通过 `"include": "./base.json"` 引用公共配置
- 每个主题文件**只包含与 base.json 不同的覆盖值**
- 修改公共配色编辑 `base.json`；修改单个主题编辑对应文件

### 语义高亮

此卷遵循 [VS Code 语义高亮规范](https://code.visualstudio.com/api/language-extensions/semantic-highlight-guide)，利用语义标记（semantic tokens）使代码着色不仅依语法而定，更依语义而变——如类名、函数名、参数名各有其色，层次分明。主题文件中的 `semanticTokenColors` 设置与语法高亮相辅相成，共成全貌。

### 无障碍考量

配色设计需参考 [WCAG 标准](https://www.w3.org/WAI/standards-guidelines/wcag/)，确保：

- 色彩对比度满足最低可读性要求
- 信息不仅依赖色彩传达（色盲友好）
- 焦点状态有明确的视觉边界

### 主题清理脚本

```bash
python scripts/apply_include_and_clean.py
```

编辑主题后运行此脚本，自动移除与 base.json 重复的值，确保只保留差异。

## 写作风格

所有面向用户的文字（README、Changelog、Git 提交、Issue 回复）均需延续统一的文言形式。遣词当有古意而不失清晰，行文当有风骨而不废技术之准确——**以文言述今事，以雅韵载实学**。

### Git Commit Message 规范

每条 commit message 遵循：`类型 简短的古典化描述 （可选）详细说明`
根据 git diff 内容自动选择诗意类型：

| 变更类型 | 诗意类型 | 意象 | 诗意描述 |
|:--------:|:--------:|:----:|:--------:|
| 新增功能 | **新章** | 开卷 | 如开新卷，添锦绣之章 |
| 修复错误 | **修误** | 补墨 | 如补墨迹，使字迹清晰 |
| 文档更新 | **文墨** | 添注 | 如添注释，增文采之韵 |
| 代码格式 | **雅致** | 调色 | 如调布局，求和谐之美 |
| 代码重构 | **重构** | 重装 | 如重装帧，使结构更佳 |
| 测试相关 | **校验** | 验色 | 如验墨色，保质量之优 |
| 构建工具 | **杂务** | 理房 | 如理文房，备笔墨纸砚 |

**示例：**

- 新增功能 → **新章: 素缃·朝霞主题初成**
- 修复错误 → **修误: 补墨迹对比之误**
- 文档更新 → **文墨: 添素缃意境之章**
- 代码格式 → **雅致: 调素缃布局之韵**

### 绝对禁止

- 网络流行语（「YYDS」「绝绝子」「家人们」）
- Emoji 堆砌（一个章节最多一次，仅用于标题前缀）
- 技术文档的干涩语气（「Fixed bug #123」）
- 过度口语化（「搞定了一个大 bug」）
- 自称「我」或「我们」——用「此卷」「此主题」
- 在非标题位置使用感叹号

### 快速参考

撰写文字前自问三问：

1. **纸上有墨吗？** ——有没有实际信息量？
2. **墨中有骨吗？** ——技术事实是否准确清晰？
3. **骨上有韵吗？** ——读起来是否有素缃特有的温润感？
